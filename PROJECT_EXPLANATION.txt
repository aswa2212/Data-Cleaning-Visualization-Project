================================================================================
 🛍️ RETAIL SALES PIPELINE & ML FORECASTING — PROJECT EXPLANATION v2.0
================================================================================

This document provides a comprehensive, plain-English breakdown of the Retail 
Sales Pipeline project. It details the system architecture, code modules, 
statistical methodologies, machine learning models, and engineering design 
choices that transform raw, messy transaction records into clean, actionable 
business intelligence.

--------------------------------------------------------------------------------
 1. PROJECT OVERVIEW & BUSINESS CONTEXT
--------------------------------------------------------------------------------
In retail operations, data arrives from multiple point-of-sale (POS) systems, 
online channels, and legacy databases. This raw transaction log is almost always 
messy—plagued by missing values, duplicates, formatting inconsistencies, negative 
returns, and outlier spikes from keying errors.

This project implements a robust, modular, configuration-driven data pipeline 
that processes this messy data and produces:
  1. A Clean, Auditable Dataset.
  2. A Data Quality Report containing an automated Data Quality (DQ) Score.
  3. Advanced Statistical Analyses (Normality, ANOVA, Chi-Square, ABC).
  4. A Predictive Machine Learning Model to forecast the next 3 months of sales.
  5. Five high-resolution dark-mode visualization dashboards.
  6. An Executive Business Summary with recommendations.

--------------------------------------------------------------------------------
 2. PIPELINE ARCHITECTURE
--------------------------------------------------------------------------------
The project follows a linear, pipeline design orchestrated by `src/main.py`. 
Each step depends on the success of the previous step:

  [config.yaml] ────────────────┐
                                ▼
  [generate_data.py] ──► raw_sales_data.csv
                                │
                                ▼
  [data_cleaner.py]  ──► 10-Step Cleaning Pipeline ──► cleaned_sales_data.csv
                                │
                                ├──► [analyzer.py]      ──► statistical_analysis.txt
                                ├──► [ml_forecaster.py] ──► sales_forecast.pkl + Feature Importances
                                ├──► [visualizer.py]    ──► 5 PNG Dashboards
                                └───────────────────────┐
                                                        ▼
                                       [report_generator.py] ──► data_quality_report.txt
                                                             ──► executive_summary.txt

--------------------------------------------------------------------------------
 3. DETAILED MODULE BREAKDOWN
--------------------------------------------------------------------------------

📁 config/config.yaml
--------------------
Centralises all tunable constants. If you want to increase the dataset size, 
adjust the outlier thresholds, modify Random Forest hyperparameters, or change 
the forecast period, you edit this file instead of modifying code.
  - Ensures flexibility and follows production software engineering principles.

📁 src/logger.py
----------------
Sets up a centralized log factory. Replacing raw `print()` statements with 
structured logging allows developers to track pipeline runs in real-time on the 
terminal while keeping persistent logs in `outputs/logs/pipeline.log`.

📁 src/generate_data.py
-----------------------
Generates a realistic, synthetic transaction dataset (3,000+ rows) containing:
  - Demographic features: Customer IDs, names, ages, loyalty tiers.
  - Geographic features: Indian cities and states mapped to regions.
  - Transactional features: Category (Electronics, Apparel, Home, etc.), items, 
    quantities, unit prices, discounts, ratings, return flags.
  - Artificially injected anomalies (nulls, duplicates, invalid dates, negative 
    quantities, rating outliers) to stress-test our data-cleaning pipeline.

📁 src/data_cleaner.py (The 10-Step Pipeline)
---------------------------------------------
Executes data preprocessing to guarantee the dataset conforms to a strict schema:
  1.  LOAD: Reads the CSV file.
  2.  DUPLICATES: Removes exact row duplicates.
  3.  DATE PARSING: Parses dates and imputes invalid formats using the median date.
  4.  STANDARDISE CATEGORICALS: normalizes strings (e.g. mapping "m", "male", 
      "MALE" -> "Male") and trims whitespace. Unknown variants are kept as-is.
  5.  INVALID VALUES: Flags negative quantities and converts them to absolute values, 
      and flags out-of-range ratings (>5.0 or <1.0) and converts them to nulls.
  6.  IMPUTE MISSING VALUES: Fills numerical missing values with medians and 
      categorical missing values with modes.
  7.  OUTLIER TREATMENT: Caps unit_price outliers using the IQR rule (Q1 - 3*IQR, 
      Q3 + 3*IQR). total_sales is NOT capped here since it is a derived column.
  8.  RECOMPUTE DERIVED COLUMNS: Recomputes total_sales as:
      `unit_price * quantity * (1 - discount)` and extracts year/month/quarter.
  9.  TYPE CASTING: Casts integers and formats categories to save memory.
  10. VALIDATION: Performs final assertions (no nulls, ratings in range, prices > 0).

📁 src/analyzer.py
------------------
Performs advanced, production-grade statistical operations:
  - Descriptive Statistics: Mean, median, skewness, and kurtosis.
  - Shapiro-Wilk Test: Assesses whether sales distributions are normal.
  - ANOVA (Analysis of Variance): Determines if average total sales differ 
    significantly across geographical regions.
  - Chi-Square Test of Independence: Tests whether payment method preference is 
    influenced by customer gender.
  - ABC Classification: Segments products into:
      - Class A: Top 70% of revenue (critical products).
      - Class B: Next 20% of revenue (moderate products).
      - Class C: Bottom 10% of revenue (low-value items).

📁 src/ml_forecaster.py
-----------------------
Builds an ML forecasting engine for sales revenue:
  - Aggregates transactions into a daily revenue time series.
  - Feature Engineering: Computes lag variables (1-day, 2-day, 7-day lags), 
    rolling averages (7-day, 30-day rolling means), and calendar features.
  - Model: Random Forest Regressor (highly robust to non-linear trend adjustments).
  - Validation: TimeSeriesSplit Cross-Validation (5-fold) to prevent data leakage.
  - Forecast: Extrapolates predictions for the next 3 months with uncertainty bands.

📁 src/visualizer.py
--------------------
Generates 5 publication-quality dark-mode dashboards using Matplotlib/Seaborn:
  - Dashboard 1: Overview Dashboard (Key business metrics, regional trends).
  - Dashboard 2: Distributions (Histograms with Kernel Density Estimation overlay).
  - Dashboard 3: Correlations (Correlation heatmap, discount vs. sales regression).
  - Dashboard 4: Seasonal Trends (Quarterly revenue splits, top products).
  - Dashboard 5: ML Insights & Product ABC (Actual vs. predicted sales, 3-month 
    forecast with confidence intervals, feature importances, and ABC classifications).

📁 src/report_generator.py
--------------------------
Generates two textual files for stakeholders:
  1. `data_quality_report.txt`: Explains every correction, lists final null counts, 
     and lists the automated Data Quality Score.
  2. `executive_summary.txt`: Business-facing summary summarizing high-level KPIs, 
     top-performing items, statistical conclusions, and ML forecast ranges.

--------------------------------------------------------------------------------
 4. ADVANCED DATA SCIENCE & ENGINEERING CONCEPTS
--------------------------------------------------------------------------------

A. Data Quality (DQ) Score Heuristic
------------------------------------
A custom mathematical score evaluates data quality out of 100:
  DQ Score = 100 - [Row Loss Deductions] - [Remaining Null Deductions] - [Issue Deductions]
    - Deductions are applied for duplicates, invalid dates, negative quantities, 
      ratings out of bounds, and schema validation failures.
    - Provides a quantitative benchmark for data governance.

B. Bug Fixes (Comparing v2.0 vs v1.0)
------------------------------------
  1. Outlier Capping Bug: In v1.0, outliers were capped on both `unit_price` and 
     `total_sales`. Capping `total_sales` directly created inconsistencies 
     because `total_sales` must be mathematically equal to `unit_price * quantity * (1-discount)`. 
     In v2.0, we only cap `unit_price` and recompute `total_sales`.
  2. Step Ordering Bug: In v1.0, outlier treatment ran before imputation. Imputing 
     nulls after clipping outliers could introduce unclipped median values, 
     re-introducing potential outliers. v2.0 runs imputation first, then caps.
  3. Safe Gender Mapping: In v1.0, unknown gender entries were silently converted 
     to NaN. In v2.0, unknown values are kept as-is, and warning logs alert 
     analysts to downstream schema anomalies.

C. Statistical Tests & Business Interpretations
-----------------------------------------------
  - ANOVA (Analysis of Variance): Used to test if the mean of a continuous variable 
    differs across multiple groups. In business, it tells us if regional sales 
    performance variations are statistically significant or just random noise.
  - Chi-Square (χ²) Independence Test: Determines if two categorical variables 
    are related. It checks if customer gender is associated with payment preferences, 
    guiding targeted digital payment promotions.
  - Shapiro-Wilk Test: Evaluates if our data comes from a normal distribution. 
    This informs whether parametric or non-parametric algorithms are appropriate.

D. ML Time-Series Forecasting
-----------------------------
  - Lags: Using historical values (e.g., Sales at t-1, t-2, t-7) as predictors.
  - Rolling Averages: Captures short-term trends (7-day mean) and long-term 
    trends (30-day mean) to smooth out weekday/weekend variations.
  - TimeSeriesSplit: Traditional K-fold CV shuffles data, which leaks future information 
    into the past. Time-series cross-validation trains on indices 0..K and tests on K+1, 
    preserving temporal ordering.
